38// The order of primary colors in the NeoPixel data stream can vary
39// among device types, manufacturers and even different revisions of
40// the same item. The third parameter to the Adafruit_NeoPixel
41// constructor encodes the per-pixel byte offsets of the red, green
42// and blue primaries (plus white, if present) in the data stream --
43// the following #defines provide an easier-to-use named version for
44// each permutation. e.g. NEO_GRB indicates a NeoPixel-compatible
45// device expecting three bytes per pixel, with the first byte
46// containing the green value, second containing red and third
47// containing blue. The in-memory representation of a chain of
48// NeoPixels is the same as the data-stream order; no re-ordering of
49// bytes is required when issuing data to the chain.
50
51// Bits 5,4 of this value are the offset (0-3) from the first byte of
52// a pixel to the location of the red color byte. Bits 3,2 are the
53// green offset and 1,0 are the blue offset. If it is an RGBW-type
54// device (supporting a white primary in addition to R,G,B), bits 7,6
55// are the offset to the white byte...otherwise, bits 7,6 are set to
56// the same value as 5,4 (red) to indicate an RGB (not RGBW) device.
57// i.e. binary representation:
58// 0bWWRRGGBB for RGBW devices
59// 0bRRRRGGBB for RGB
60
61// RGB NeoPixel permutations:
62// Offset: W R G B
63#define NEO_RGB ((0 << 6) | (0 << 4) | (1 << 2) | (2))
64#define NEO_RBG ((0 << 6) | (0 << 4) | (2 << 2) | (1))
65#define NEO_GRB ((1 << 6) | (1 << 4) | (0 << 2) | (2)) // GRB is very common
66#define NEO_GBR ((2 << 6) | (2 << 4) | (0 << 2) | (1))
67#define NEO_BRG ((1 << 6) | (1 << 4) | (2 << 2) | (0))
68#define NEO_BGR ((2 << 6) | (2 << 4) | (1 << 2) | (0))
