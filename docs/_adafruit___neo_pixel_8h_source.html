73// The order of primary colors in the NeoPixel data stream can vary among
74// device types, manufacturers and even different revisions of the same
75// item. The third parameter to the Adafruit_NeoPixel constructor encodes
76// the per-pixel byte offsets of the red, green and blue primaries (plus
77// white, if present) in the data stream -- the following #defines provide
78// an easier-to-use named version for each permutation. e.g. NEO_GRB
79// indicates a NeoPixel-compatible device expecting three bytes per pixel,
80// with the first byte transmitted containing the green value, second
81// containing red and third containing blue. The in-memory representation
82// of a chain of NeoPixels is the same as the data-stream order; no
83// re-ordering of bytes is required when issuing data to the chain.
84// Most of these values won't exist in real-world devices, but it's done
85// this way so we're ready for it (also, if using the WS2811 driver IC,
86// one might have their pixels set up in any weird permutation).
87
88// Bits 5,4 of this value are the offset (0-3) from the first byte of a
89// pixel to the location of the red color byte. Bits 3,2 are the green
90// offset and 1,0 are the blue offset. If it is an RGBW-type device
91// (supporting a white primary in addition to R,G,B), bits 7,6 are the
92// offset to the white byte...otherwise, bits 7,6 are set to the same value
93// as 5,4 (red) to indicate an RGB (not RGBW) device.
94// i.e. binary representation:
95// 0bWWRRGGBB for RGBW devices
96// 0bRRRRGGBB for RGB
97
98// RGB NeoPixel permutations; white and red offsets are always same
99// Offset: W R G B
100#define NEO_RGB ((0 << 6) | (0 << 4) | (1 << 2) | (2))
101#define NEO_RBG ((0 << 6) | (0 << 4) | (2 << 2) | (1))
102#define NEO_GRB ((1 << 6) | (1 << 4) | (0 << 2) | (2))
103#define NEO_GBR ((2 << 6) | (2 << 4) | (0 << 2) | (1))
104#define NEO_BRG ((1 << 6) | (1 << 4) | (2 << 2) | (0))
105#define NEO_BGR ((2 << 6) | (2 << 4) | (1 << 2) | (0))

137// Add NEO_KHZ400 to the color order value to indicate a 400 KHz device.
138// All but the earliest v1 NeoPixels expect an 800 KHz data stream, this is
139// the default if unspecified. Because flash space is very limited on ATtiny
140// devices (e.g. Trinket, Gemma), v1 NeoPixels aren't handled by default on

249 bool canShow(void)
250 {
251 // It's normal and possible for endTime to exceed micros() if the
252 // 32-bit clock counter has rolled over (about every 70 minutes).
253 // Since both are uint32_t, a negative delta correctly maps back to
254 // positive space, and it would seem like the subtraction below would
255 // suffice. But a problem arises if code invokes show() very
256 // infrequently...the micros() counter may roll over MULTIPLE times in
257 // that interval, the delta calculation is no longer correct and the
258 // next update may stall for a very long time. The check below resets
259 // the latch counter if a rollover has occurred. This can cause an
260 // extra delay of up to 300 microseconds in the rare case where a
261 // show() call happens precisely around the rollover, but that's
262 // neither likely nor especially harmful, vs. other code that might
263 // stall for 30+ minutes, or having to document and frequently remind
264 // and/or provide tech support explaining an unintuitive need for
265 // show() calls at least once an hour.
266 uint32_t now = micros();
267 if (endTime > now)
268 {
269 endTime = now;
270 }
271 return (now - endTime) >= 300L;
272 }
